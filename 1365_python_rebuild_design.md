# 1365 봉사 공고 탐색 서비스 재구축 설계서

## 1. 문서 목적
이 문서는 기존 **JavaScript 기반 정적 페이지 프로젝트를 폐기하고**, **FastAPI + React + PostgreSQL 기반 서비스**로 1365 봉사 공고 탐색 서비스를 다시 구현하기 위한 설계 문서다.

이 문서는 아래 4가지를 명확히 하기 위해 작성한다.

1. 무엇을 만들지 정의한다.
2. 어떤 기술 스택으로 만들지 확정한다.
3. 어떤 순서로 구현할지 정한다.
4. 어디까지를 MVP로 볼지 구분한다.

이 문서는 바이브 코딩 중 방향이 흔들리지 않도록, 기능 범위와 제외 범위를 최대한 명확하게 적는다.

---

## 2. 프로젝트 한 줄 정의
전국 봉사 공고 데이터를 수집·저장하고, 사용자가 **조건 검색 탭** 또는 **챗봇 추천 탭**을 통해 원하는 공고를 탐색할 수 있는 **FastAPI + React 기반 웹서비스**를 구축한다.

---

## 3. 현재 상황과 재구축 방향

### 3.1 현재 상황
- 기존 프로젝트는 JavaScript 기반 정적 페이지 중심이다.
- 현재는 서울 데이터 중심으로 동작한다.
- 기능은 공고 수집 및 리스트업에 가깝다.
- DB 중심 구조가 아니라 대용량 전국 데이터를 다루기 어렵다.

### 3.2 재구축 방향
- 기존 구조를 유지 보수하지 않는다.
- **Python 백엔드 + React 프론트엔드 + PostgreSQL DB** 구조로 다시 만든다.
- 데이터 범위를 **서울 중심 → 전국 단위**로 확장한다.
- 메인 화면은 **탭 분리 UI**로 단순하게 설계한다.
  - 탭 1: 조건 검색
  - 탭 2: 챗봇 추천
- 검색과 추천의 핵심은 **SQL 기반 조회**다.
- 챗봇은 자연어를 조건으로 해석해 SQL 조회를 수행하는 인터페이스다.
- RAG는 이번 재구축의 필수 범위에 넣지 않는다.

---

## 4. 핵심 원칙

### 원칙 1. 서비스의 중심은 검색이다
이 서비스의 본질은 봉사 공고 탐색이다.
따라서 가장 중요한 것은 정확한 데이터 저장, 필터링, 리스트업, 정렬, 페이징이다.
챗봇은 메인 로직이 아니라 검색을 보조하는 입력 방식이다.

### 원칙 2. SQL이 중심이다
전국 단위 공고를 다루기 위해서는 DB가 필수다.
지역, 날짜, 유형, 모집 상태, 정렬, 통계, 중복 제거를 SQL 중심으로 해결한다.

### 원칙 3. 챗봇은 범용 AI가 아니다
챗봇은 자유 대화형 비서가 아니라, 사용자의 자연어 요구를 검색 조건으로 변환하고 추천 결과를 정리해주는 **자연어 검색 인터페이스**다.

### 원칙 4. UI는 단순해야 한다
한 화면에 챗봇과 조건 검색을 동시에 크게 보여주지 않는다.
메인 화면은 **탭 분리 구조**를 사용한다.

### 원칙 5. MVP를 먼저 완성한다
처음부터 RAG, 개인화 추천, 유사 공고 추천까지 확장하지 않는다.
먼저 전국 데이터 저장, 조건 검색, 챗봇 추천의 3가지를 완성한다.

## 5. 최종 기술 스택 확정

### 5.1 프론트엔드
- **React**
- 추천 라이브러리: **Vite + React + TypeScript**
- 스타일링: **Tailwind CSS** 권장
- 상태 관리: 초기에는 React 기본 state 또는 Zustand
- API 통신: fetch 또는 axios

선정 이유:
- 탭 분리 UI를 만들기 쉽다.
- 조건 검색과 챗봇을 컴포넌트 단위로 분리하기 좋다.
- 결과 리스트, 로딩 상태, 에러 상태, 탭 전환 관리가 편하다.
- 추후 UI 완성도 향상에 유리하다.

### 5.2 백엔드
- **Python**
- **FastAPI**
- **SQLAlchemy**
- **Pydantic**
- 필요 시 Alembic

선정 이유:
- API 개발이 빠르고 명확하다.
- 챗봇, SQL, 배치 수집, 추후 RAG 확장이 자연스럽다.
- OpenAPI 문서가 자동 생성되어 테스트가 쉽다.
- Python 생태계로 데이터 수집과 정제를 통일할 수 있다.

### 5.3 데이터베이스
- **PostgreSQL**

선정 이유:
- 대용량 공고 저장과 조회에 안정적이다.
- 조건 필터링과 정렬에 강하다.
- 추후 pgvector 등 확장 가능성이 있다.

### 5.4 수집/배치
- Python script 또는 FastAPI와 분리된 수집 모듈
- 초기에는 별도 Python 스크립트로 시작
- 이후 cron, GitHub Actions, scheduler 등으로 확장 가능

### 5.5 LLM/챗봇
- 초기 구현은 **키워드/룰 기반 + 선택적 LLM 보조**
- 자연어 조건 추출이 불안정하면 LLM API를 붙인다.
- 그러나 추천 결과 조회는 반드시 SQL 기반으로 한다.

## 6. 사용자 시나리오

### 시나리오 A. 조건 검색 사용자
1. 사용자가 홈에 진입한다.
2. `조건 검색` 탭을 선택한다.
3. 지역, 날짜, 시간대, 활동 유형 등의 조건을 설정한다.
4. 검색 버튼을 누른다.
5. 조건에 맞는 공고 리스트가 나온다.
6. 사용자는 공고 카드에서 상세 정보나 원문 링크를 확인한다.

### 시나리오 B. 챗봇 추천 사용자
1. 사용자가 홈에 진입한다.
2. `챗봇 추천` 탭을 선택한다.
3. 자연어 또는 키워드로 원하는 조건을 입력한다.
   - 예: `서울에서 주말 오전에 가능한 교육봉사 추천해줘`
4. 챗봇이 지역, 날짜, 시간대, 유형 등의 조건을 해석한다.
5. FastAPI가 SQL 기반으로 후보 공고를 조회한다.
6. 추천 결과 3~5개와 추천 이유를 함께 보여준다.

## 8. 화면 설계

## 8.1 홈 화면

### 헤더
- 서비스 이름
- 짧은 설명
  - 예: `조건 검색과 챗봇 추천으로 봉사 공고를 쉽게 찾는 서비스`

### 본문

  - `조건 검색`
  - `챗봇 추천`

탭을 클릭하면 해당 탭의 내용만 렌더링한다.

---

## 8.2 탭 1: 조건 검색 화면

### 목적
사용자가 명시적으로 조건을 설정하고, 원하는 공고를 직접 검색한다.

### 화면 구성

#### 검색 필터 영역
- 시/도 선택
- 시/군/구 선택
- 봉사 날짜 또는 기간
- 시간대
- 활동 유형
- 봉사 대상
- 모집 상태
- 정렬 기준
- 검색 버튼
- 초기화 버튼

#### 결과 영역
각 공고 카드는 아래 정보를 포함한다.
- 제목
- 기관명
- 지역
- 봉사 날짜
- 시간
- 활동 유형
- 모집 상태
- 상세 보기 버튼
- 원문 링크 버튼

### UX 규칙
- 검색 전 기본 결과를 보여줄지 여부를 결정한다.
  - 권장: 최근 공고 일부 표시
- 결과가 없으면 빈 상태 메시지를 보여준다.
- 로딩 중 상태를 명확히 보여준다.
- 필터는 사용자가 조작하기 쉽게 그룹화한다.

---

## 8.3 탭 2: 챗봇 추천 화면

### 목적
사용자가 자연어로 조건을 입력하면 공고를 추천받는다.

### 화면 구성

#### 입력 영역
- 채팅 입력창
- 전송 버튼
- 예시 질문 버튼 또는 placeholder

예시 질문:
- `서울에서 주말에 할 수 있는 봉사 추천해줘`
- `대학생이 하기 쉬운 단기 봉사 5개 보여줘`
- `아동 대상 교육봉사 위주로 추천해줘`

#### 응답 영역
응답은 아래 순서로 보여준다.
1. 해석한 조건 요약
2. 추천 공고 리스트
3. 각 공고별 추천 이유 한 줄

### UX 규칙
- 장문 대화보다 짧고 목적형 질의응답에 집중한다.
- 추천 결과는 3~5개로 제한한다.
- 조건이 모호하면 재입력을 유도한다.
- `이 공고 더 보여줘`, `비슷한 조건 다시 추천` 같은 후속 질문은 후순위로 둔다.

## 9. 기능 범위 정의

## 9.1 반드시 구현할 기능

### A. 전국 데이터 수집 및 저장
- 전국 공고를 수집할 수 있어야 한다.
- 공고를 DB에 저장해야 한다.
- 중복 공고를 식별하고 중복 저장을 막아야 한다.

### B. 조건 검색
- 사용자가 필터 조건을 설정할 수 있어야 한다.
- SQL 기반으로 검색 결과를 반환해야 한다.
- 정렬과 페이징 또는 개수 제한이 가능해야 한다.

### C. 챗봇 추천
- 자연어/키워드 입력을 받을 수 있어야 한다.
- 입력에서 지역, 날짜, 유형 등의 조건을 추출할 수 있어야 한다.
- 추출한 조건으로 SQL 조회를 수행해야 한다.
- 추천 결과와 추천 이유를 보여줘야 한다.

### D. 공고 상세 확인
- 공고의 세부 정보를 확인할 수 있어야 한다.
- 원문 링크로 이동할 수 있어야 한다.

---

## 9.2 있으면 좋은 기능
- 결과 리스트에서 빠른 비교
- 북마크
- 최근 검색어
- 추천 이유 개선
- 모바일 대응 강화

---

## 9.3 이번 범위에서 제외할 기능
- 본문 기반 RAG QA
- 사용자 맞춤 개인화 추천
- 로그인/회원 시스템
- 복잡한 대화 메모리
- 관리자 대시보드 고도화
- 고급 통계 페이지

제외 이유:
이번 목표는 **전국 데이터 + 조건 검색 + 챗봇 추천**을 안정적으로 완성하는 것이다.

## 10. 데이터 설계

## 10.1 핵심 테이블: volunteer_posts

권장 컬럼 예시:

- `id`: 내부 PK
- `source_post_id`: 원본 공고 식별자
- `title`: 공고 제목
- `organization_name`: 기관명
- `province`: 시/도
- `city_district`: 시/군/구
- `recruit_start_date`: 모집 시작일
- `recruit_end_date`: 모집 마감일
- `volunteer_date_start`: 봉사 시작일
- `volunteer_date_end`: 봉사 종료일
- `time_text`: 시간 텍스트 원문
- `start_time`: 시작 시간
- `end_time`: 종료 시간
- `activity_type`: 활동 유형
- `target_text`: 봉사 대상
- `status`: 모집 상태
- `description`: 상세 설명
- `source_url`: 원문 링크
- `collected_at`: 수집 시각
- `updated_at`: 최신 업데이트 시각

### 인덱스 권장
- `source_post_id`
- `province`
- `city_district`
- `volunteer_date_start`
- `activity_type`
- `status`
- `(province, city_district)` 복합 인덱스

---

## 10.2 중복 처리 원칙
중복 판별 우선순위:
1. 원본 공고 ID가 있으면 그것을 기준으로 upsert
2. 없으면 제목 + 기관명 + 날짜 + 지역 조합으로 후보 판별

원칙:
- 같은 공고는 중복 저장하지 않는다.
- 변경 사항이 있으면 update 처리한다.

---

## 11.3 챗봇 추천 API
### `POST /api/chat/recommend`
목적: 자연어 입력을 받아 추천 결과 반환

입력 예시:
```json
{
  "message": "서울에서 주말 오전에 가능한 교육봉사 추천해줘"
}
```

응답 예시:
```json
{
  "parsed_conditions": {
    "province": "서울",
    "day_type": "주말",
    "time_slot": "오전",
    "activity_type": "교육봉사"
  },
  "results": [
    {
      "id": 1,
      "title": "청소년 학습 멘토링 봉사",
      "reason": "서울 지역이며 주말 오전 활동 조건과 잘 맞습니다."
    }
  ]
}
```

## 12. 챗봇 추천 로직 설계

## 12.1 1차 목표
챗봇은 아래 흐름으로 동작한다.

1. 사용자 입력 수신
2. 조건 추출
3. SQL 조회 조건 생성
4. 후보 공고 검색
5. 상위 결과 선택
6. 추천 이유 생성
7. 결과 반환

---

## 12.2 조건 추출 대상
최소한 아래 항목을 뽑는다.

- 주말/평일

- 활동 유형
- 대상

## 12.3 구현 방식
### 우선순위 1
룰 기반/키워드 기반 파서
- 빠르고 통제 가능
- 디버깅 쉬움
- MVP에 적합

### 우선순위 2
필요 시 LLM 보조
- 사용자의 표현이 다양해질 때 도입
- 다만 LLM은 **조건 추출 보조용**이다.
- 최종 추천 결과는 반드시 SQL 조회 결과를 기반으로 해야 한다.

---

## 12.4 추천 이유 생성 방식
처음에는 규칙 기반으로 충분하다.
예:
- `서울 지역 조건과 일치합니다.`
- `주말 오전 활동이 가능해 요청한 조건에 맞습니다.`
- `교육봉사 유형으로 분류된 공고입니다.`

MVP에서는 이유를 간단하고 신뢰 가능하게 유지한다.

## 16. MVP 정의

### MVP 범위
- FastAPI + React + PostgreSQL 구조 완성
- 전국 공고 DB 저장
- 조건 검색 탭 동작
- 챗봇 추천 탭 동작
- 공고 상세 보기 가능

### MVP에서 요구하지 않는 것
- 완벽한 자연어 이해
- 복잡한 대화 맥락 유지
- 본문 기반 RAG 질의응답
- 사용자 개인 맞춤 추천

MVP의 핵심은 **검색과 추천이 실제로 동작하는 것**이다.

## 18. 리스크와 대응

### 리스크 1. 전국 데이터 수집 난이도
대응:
- 먼저 구조를 만들고, 데이터 수집 범위를 점진적으로 확장한다.
- 초기에는 일부 지역 샘플로 시작해도 된다.

### 리스크 2. 챗봇 정확도 부족
대응:
- 룰 기반 파서를 먼저 만든다.
- 필요 시 LLM을 조건 추출에만 제한적으로 도입한다.

### 리스크 3. 화면이 복잡해짐
대응:
- 탭 분리 구조를 유지한다.
- 조건 검색과 챗봇을 한 화면에 동시에 크게 노출하지 않는다.

### 리스크 4. 범위가 너무 커짐
대응:
- RAG와 개인화는 제외한다.
- MVP 완성 후 추가한다.

## 19. 최종 구현 목표 요약

이 프로젝트의 최종 목표는 아래와 같다.

1. 전국 봉사 공고를 저장할 수 있는 DB 기반 구조를 만든다.
2. 사용자가 조건 검색 탭에서 직접 공고를 찾을 수 있게 한다.
3. 사용자가 챗봇 추천 탭에서 자연어로 공고를 추천받을 수 있게 한다.
4. 전체 서비스를 FastAPI + React 기반으로 재구축한다.

### 우선 작업

- UI 욕심내다가 검색 기능이 늦어지기

## 21. 포트폴리오용 설명 문장

### 한 줄 소개
전국 봉사 공고 데이터를 수집·구조화하고, SQL 기반 조건 검색과 자연어 챗봇 추천을 결합한 봉사 공고 탐색 웹 서비스를 FastAPI와 React로 재구축했다.

### 문제 정의 문장
기존 봉사 공고 탐색은 지역, 일정, 활동 유형 등 다양한 조건을 빠르게 반영하기 어렵고, 사용자가 원하는 공고를 찾는 과정이 번거롭다. 이를 해결하기 위해 전국 공고 데이터를 구조화하여 저장하고, 직접 조건 검색과 자연어 추천을 함께 제공하는 탐색 서비스를 설계했다.

### 핵심 구현 문장
FastAPI 백엔드와 PostgreSQL을 활용해 전국 공고 데이터를 저장·조회하는 API를 구축하고, React 프론트엔드에서 조건 검색 탭과 챗봇 추천 탭을 분리해 사용자 경험을 단순화했다.

---

## 22. 결론
이 프로젝트는 단순한 봉사 공고 웹페이지가 아니라,
**전국 공고 데이터 수집 + SQL 기반 탐색 + 자연어 추천 인터페이스**를 포함한 서비스형 프로젝트로 확장하는 것이 목표다.

이번 재구축에서 가장 중요한 것은 아래 순서를 지키는 것이다.

**DB 구축 → 조건 검색 완성 → 챗봇 추천 추가 → UI 정리**

이 순서를 지키면 범위가 흔들리지 않고, 포트폴리오용으로도 설명이 명확한 프로젝트를 만들 수 있다.
